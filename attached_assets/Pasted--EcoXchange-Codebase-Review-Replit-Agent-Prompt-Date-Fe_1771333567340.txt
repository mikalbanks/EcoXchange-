## Part 1: Codebase Review — Key Errors & Issues

### CRITICAL — Security Vulnerabilities

**1. SHA-256 Password Hashing (storage.ts:14-16)**  
You’re using raw `crypto.createHash("sha256")` for password hashing. This is fundamentally broken for production — SHA-256 is a fast hash, meaning it’s trivially brute-forceable. Any investor-facing or securities-related platform needs bcrypt, scrypt, or Argon2 at minimum. This would be an instant red flag for any securities counsel or technical due diligence.

```typescript
// CURRENT (INSECURE)
function hashPassword(password: string): string {
  return createHash("sha256").update(password).digest("hex");
}
```

**2. Hardcoded Session Secret (routes.ts:19)**  
`"ecoxchange-demo-secret"` as a fallback session secret is a session hijacking vulnerability. In production this means any attacker who reads the source code (which is in your Replit) can forge session cookies.

**3. Webhook Signature Validation is Optional (routes.ts:174)**  
The Persona webhook handler silently proceeds if `PERSONA_WEBHOOK_SECRET` is not set. An attacker could forge webhook events to mark any user as KYC-verified. For a 506(c) platform where accredited investor verification is a legal requirement, this is a compliance-breaking vulnerability.

**4. No CSRF Protection**  
Express sessions with cookies but no CSRF token validation. Any malicious site could submit requests on behalf of an authenticated user.

**5. No Rate Limiting**  
No rate limiting on auth endpoints (`/api/auth/login`, `/api/auth/signup`). Brute-force attacks are trivial.

-----

### HIGH — Architectural Issues

**6. MemoryStore for Sessions and All Data (storage.ts:196)**  
The entire application runs on in-memory Maps. Every server restart wipes all users, projects, interests, and approval logs. This is expected for a prototype, but the schema.ts file defines full Postgres table schemas via Drizzle that are never actually used — meaning you have dead ORM code that creates confusion about what the actual persistence layer is.

**7. Package Name Mismatch (package.json:2)**  
`"name": "rest-express"` — this is a Replit template default, not your project name. Minor, but signals to anyone reviewing (accelerators, advisors) that the codebase was scaffolded rather than intentionally architected.

**8. Express 5 Beta (package.json:52)**  
You’re on `express@^5.0.1` which is still not stable. Express 5 has breaking changes from v4 in error handling and route parameter behavior. Given that this is a securities platform, using a beta web framework is unnecessary risk.

**9. Tailwind v3 + v4 Conflict (package.json:84,99)**  
You have `@tailwindcss/vite@^4.1.18` (Tailwind v4 plugin) alongside `tailwindcss@^3.4.17`. The Vite plugin is for Tailwind v4 but your config and usage patterns are v3. This creates build inconsistencies and will cause silent CSS failures when dependencies auto-update.

**10. `persona` npm Package is Wrong (package.json:61)**  
`"persona": "^5.5.0"` — this is the `persona` npm package, NOT Persona’s `@persona-id/client` or their embedded SDK. The Persona integration in routes.ts correctly uses fetch against their REST API, but this dependency is unused dead code that could confuse future developers.

**11. staleTime: Infinity (queryClient.ts:49)**  
All React Query queries are set to never go stale. This means once data is fetched, it will never auto-refresh. If an admin approves a project, the investor’s deal list won’t update until they hard refresh. For a demo this is fine, but it creates a misleading UX where actions appear to have no effect.

-----

### MEDIUM — Data Model & Business Logic Gaps

**12. No Yield/Distribution Model in Schema**  
The NorthStar identifies energy-to-yield infrastructure as the “most defensible technology advantage” (Pillar 2), but the schema has zero tables for: energy production data, PPA terms, revenue ingestion, yield calculations, distribution records, or investor payout history. The “Yield Simulation Dashboard” mentioned in the NorthStar doesn’t actually exist in the codebase.

**13. Capital Stack is Oversimplified (storage.ts:187-192)**  
`equityNeeded = totalCapex - taxCreditEstimated` — this ignores debt, developer equity, reserves, construction financing, and operating expenses. For a platform positioning itself to sophisticated accredited investors, this formula will immediately undermine credibility. Real project finance capital stacks have 4-6 layers.

**14. Readiness Score Computed on Empty Docs (storage.ts:297-300)**  
Seed data computes readiness scores for Project 1 with empty document and checklist arrays, even though it then creates documents and marks checklist items as UPLOADED. The displayed score doesn’t match the actual project state. The score is never recomputed after seeding.

**15. No Foreign Key Constraints**  
The schema defines `developerId`, `projectId`, `investorId` as plain varchar columns with no `.references()` calls. Drizzle supports FK constraints but none are used. This means orphaned records are possible and there’s no referential integrity.

**16. Project Size Mismatch with NorthStar**  
Seed data has a 25MW project at $28.5M capex. The NorthStar explicitly targets $1M-$5M projects that are “too small for institutional project finance.” A $28.5M project is exactly the type that traditional project finance DOES fund. Your demo data contradicts your thesis.

-----

### LOW — Code Quality & DX

**17. `any` Types Everywhere (routes.ts)**  
`req: any, res: any, next: any` throughout all route handlers. TypeScript is essentially turned off for the entire server layer.

**18. Duplicated Utility Functions**  
`formatCurrency`, `formatTimeline`, `formatDate` are duplicated across investor/dashboard.tsx, investor/deal-room.tsx, and other pages rather than extracted to a shared utils file.

**19. No Error Boundaries**  
No React error boundaries. Any uncaught rendering error will white-screen the entire app.

**20. queryKey URL Joining (queryClient.ts:32)**  
`queryKey.join("/")` means the query key `["/api/investor/deals", id]` produces `/api/investor/deals/[id]` — this only works if keys are structured as `[baseUrl, ...pathSegments]`. It’s fragile and non-standard for React Query.

```typescript
export function computeMonthlyRevenue(
  productionMWh: number,
  ppaRate: number,
  opexPerMWh: number = 5.0, // $5/MWh default O&M
  debtServiceMonthly: number = 0,
  reserveRate: number = 0.05 // 5% reserve
): {
  grossRevenue: number;
  operatingExpenses: number;
  debtService: number;
  reserves: number;
  netDistributable: number;
} {
  const grossRevenue = productionMWh * ppaRate;
  const operatingExpenses = productionMWh * opexPerMWh;
  const reserves = grossRevenue * reserveRate;
  const netDistributable = Math.max(
    grossRevenue - operatingExpenses - debtServiceMonthly - reserves,
    0
  );
  return { grossRevenue, operatingExpenses, debtService: debtServiceMonthly, reserves, netDistributable };
}

export function computeInvestorDistribution(
  netDistributable: number,
  investorOwnershipPct: number, // e.g., 0.15 for 15%
  holdbackRate: number = 0.0 // optional holdback
): {
  grossAmount: number;
  holdbackAmount: number;
  netAmount: number;
} {
  const grossAmount = netDistributable * investorOwnershipPct;
  const holdbackAmount = grossAmount * holdbackRate;
  const netAmount = grossAmount - holdbackAmount;
  return { grossAmount, holdbackAmount, netAmount };
}

export function generateSimulatedProduction(
  capacityMW: number,
  month: number, // 1-12
  state: string // for regional solar resource estimation
): { productionMWh: number; capacityFactor: number } {
  // Monthly capacity factors by region (simplified)
  const solarProfiles: Record<string, number[]> = {
    "Texas":    [0.18, 0.20, 0.23, 0.25, 0.27, 0.28, 0.27, 0.26, 0.24, 0.22, 0.19, 0.17],
    "Arizona":  [0.22, 0.24, 0.27, 0.29, 0.31, 0.30, 0.28, 0.27, 0.28, 0.26, 0.23, 0.21],
    "Nevada":   [0.20, 0.23, 0.26, 0.28, 0.30, 0.31, 0.29, 0.28, 0.27, 0.24, 0.21, 0.19],
    "California":[0.21, 0.23, 0.26, 0.28, 0.29, 0.28, 0.27, 0.27, 0.26, 0.24, 0.22, 0.20],
  };
  const defaultProfile = [0.16, 0.18, 0.21, 0.23, 0.25, 0.26, 0.25, 0.24, 0.22, 0.20, 0.17, 0.15];
  const profile = solarProfiles[state] || defaultProfile;
  const capacityFactor = profile[month - 1];
  const hoursInMonth = 730; // average
  const productionMWh = capacityMW * capacityFactor * hoursInMonth;
  return { productionMWh: Math.round(productionMWh * 100) / 100, capacityFactor };
}
```
